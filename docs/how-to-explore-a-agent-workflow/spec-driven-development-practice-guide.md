# Spec-Driven 开发落地实践指南

Spec-Driven 开发落地实践指南
Spec-Driven Development（规范驱动开发）强调“先规范，后编码”，通过结构化的产品需求文档（PRD）和验收测试场景，引导 AI 助手严格按规范实现代码。下面将基于 GitHub Spec Kit 工具和既有调研成果，介绍如何在前端为主（含部分 API）的项目中实践这一方法。
## 1. 标准 PRD 模板（Gherkin-first 方法）
**PRD 模板概览：**采用结构清晰的 Markdown 文档，包括以下核心部分：
* 用户场景与测试 (User Scenarios & Testing)：按优先级列出用户故事。每个用户故事描述用户 “想要做什么” 及其业务价值，并包含验收场景。验收场景使用 Gherkin 语法（Given/When/Then）编写，可独立测试每个用户故事是否满足要求。重点覆盖前端交互流程的正常路径和关键边界情况，确保每个用户故事的结局都清晰可验收。 验收场景示例：
    * Given 用户打开设置页面 When 点击“Goals”标签 Then 显示其当前每日目标
    * Given 用户输入无效数据（如负数） When 尝试保存 Then 出现错误提示
* 功能性需求 (Requirements)：分条列出系统需满足的功能和业务逻辑要求，通常用编号标识（如 FR-001、FR-002）。这些需求可以对应上述用户场景，描述前端与后端的交互契约或业务规则。例如：
    * FR-001: 系统必须允许用户编辑并保存每日目标，提供数值输入框来修改“热量”“蛋白质”等指标
    * FR-002: 系统必须验证目标输入（只接受非负数字），对无效数据显示明确的错误信息
    * FR-003: 前端保存目标时，应调用后端 API 接口**POST /api/goals更新目标数据，并正确处理返回结果（例如返回状态和更新时间戳） (接口契约示例：请求体包含calories*, **protein字段…，成功时后端返回JSON包含status: "success"*及更新后的时间等)
* 成功标准 (Success Criteria)：定义可量化验证功能成功与否的指标。这些标准聚焦用户视角的可测结果，不涉及实现细节。例如：页面在保存目标后即时刷新并反映新数据；错误提示信息在0.5秒内出现；或依照业务KPI定义的成功条件（如100%用例通过验收测试）。成功标准应尽量具体、可测量，例如“目标更新后，用户日记页面的进度条实时*反映新的目标值”*，确保每条标准都能被客观验证。
上述模板以验收测试场景（Gherkin）为先导，贯穿前端交互与后端业务逻辑需求，实现*“规格即验收”*。这一结构使规范文档成为代码实现的源头，使开发团队和 AI 助手始终以统一的行为描述为准绳。通过将关键行为用 Given/When/Then 场景描述，确保正常路径和边界条件都被覆盖，形成完整的需求边界。
## 2. Prompt 模板：从自然语言反馈到规范文档与验收测试
在实践中，可以借助 Prompt 模板将用户或产品经理的自然语言需求转化为结构化的规范文档（spec.md）以及对应的 Gherkin 验收测试场景。以下提供一个 Prompt 模板示例，可用于指导 AI 助手生成符合上述 PRD 模板的文档：
**系统角色**：你是一个“Spec-Driven”产品助理，擅长将用户反馈转化为产品规范和验收测试。

**任务**：根据以下自然语言反馈，编写一份结构化的产品需求文档（spec.md）。文档应包含：
-**用户故事**：清晰描述用户需求场景，每个故事附带“为什么需要此功能”的背景。
-**验收场景**：针对每个用户故事，给出至少1-2个验收测试场景，使用“Given-When-Then”格式描述正常路径（happy path）和重要边界情况。
-**功能性需求**：提炼关键的功能和业务逻辑要求，使用 FR-编号列表表示。
-**成功标准**：定义功能完成后的可衡量指标或期望结果。

**注意**：
- 侧重描述“**是什么**”和“**为什么**”，避免涉及具体技术实现或代码细节。
- 所有验收场景使用英文的 Given/When/Then 格式编写，确保表述可测试且无二义性。
- 如果反馈中有不明确的地方，请合理假设并在文档中标注需澄清的点。

**用户反馈示例**：
我们希望在用户的“设置”页面新增“每日营养目标”管理功能。用户需要能够查看当前的卡路里、蛋白质等每日目标，并可以修改这些数值。修改时需要验证输入的有效性，例如不允许负数或非数字。同时，希望有直观的提示告诉用户修改是否成功。

上述 Prompt 模板引导 AI 按照预定义的结构输出规范文档，并强调包含正向路径和边界场景的验收测试。例如，上面的反馈会令 AI 产出一个包含“查看和编辑每日营养目标”的用户故事，附带相应的 Gherkin 场景（如查看目标显示正确、输入无效数据时提示错误）以及功能需求列表和成功标准等内容。通过这种方式，团队可以将松散的需求描述转化为清晰严谨的 spec.md 文档，为后续开发奠定基础。
## 3. Spec-Driven Product Agent 工作流设计
采用 Spec-Driven 方法的产品开发流程，是一个**从需求反馈到绿色构建（Green Build）**的闭环。具体包括以下阶段（每一步都有人工审核介入，确保质量）：
1. Feedback（反馈需求）→ Specify（生成规范）：当收到新的产品反馈或需求时，首先由团队和产品代理（AI 助手）协作撰写/更新规范文档。通过 /specify 阶段（或上述 Prompt 模板），明确要构建什么以及为什么，聚焦于功能动机和业务需求，不提前决策技术方案。AI 会根据输入自动生成初步的 spec.md（PRD），其中包含用户故事、验收标准、业务规则等。**人工干预点：**团队仔细审阅该规范，使用 /clarify 提示AI澄清任何含糊之处（如有），补充遗漏的细节，然后确认规范。
2. Spec（规范确定）→ Plan（技术方案）：在规格明确后，进入技术规划阶段。通过 /plan，AI 根据规范和项目既定原则（如架构、技术栈约束）生成技术实施计划。Plan 文档（plan.md）描述“如何实现”：包括前端/后端框架选型、组件设计、数据模型、接口契约和依赖的服务等。例如，会检查规范要求是否需要新的 API，并在计划中定义接口和数据合同（如请求/响应结构）。**人工干预点：**开发人员审核 plan.md，确保技术方案符合团队约定（通过对照 constitution.md 项目宪章进行验证）且可行，然后再进入下一阶段。
3. Plan（技术方案）→ Tasks（任务拆解）：确认计划后，让 AI 将规范和方案分解为具体的开发任务清单。通过 /tasks，产生 tasks.md 列出逐项任务，通常每个任务是一个小的可验证单元。任务列表会标明优先级、关联的用户故事编号，以及任务类型（如编码、测试、文档）等。例如，Tasks 可能包括“实现某前端组件”、“新增后端 API 方法”、“编写相应单元测试”等，并可能提示需要人工验证的步骤。**人工干预点：**团队应审查任务拆解是否合理完整，顺序是否合适，并调整不合理的任务描述。
4. Tasks（任务清单）→ Implementation（实现）：在任务清单确定后，AI 助手会按照任务逐条实现代码（这一阶段相当于进入开发环节）。可以通过 /implement 命令让 AI 顺序执行 tasks.md 列出的任务，实现前端界面和相关后端逻辑。每完成一个任务，代码变更都会应用并局部验证，然后继续下一个任务。这种逐步实现确保每个小步都符合规范要求，降低整体出错风险。**人工干预点：**开发人员在每个任务完成后进行代码审查和测试，必要时修正 AI 实现的偏差，确保逐项任务都达到验收标准。由于每项任务都源自明确的规范片段，代码审核更为高效且有据可依。
5. Implementation（实现）→ Acceptance（验收交付）：当所有任务代码实现完毕，就进入验收环节。运行完整的测试套件或专门的验收验证脚本，确保所有验收场景（Given/When/Then）测试通过，构建为绿色 (Green Build)[](https:///blog/posts/from-spec-to-shipping-how-developers-implement-features-with-ai-driven-workflows#:~:text=,)[](https:///blog/posts/from-spec-to-shipping-how-developers-implement-features-with-ai-driven-workflows#:~:text=%23 SC,)。Spec Kit 等工具甚至会生成验证脚本（如 verify_implementation.sh）来自动检查代码改动、运行相关测试并核对验收标准。团队需查看这些自动验证结果，并进行任何需要的手工测试（例如UI视觉检查）。当验收标准全部满足，即可合并代码并交付功能。若有未通过的测试或未满足的验收条件，则需回到相应阶段（可能是补充规范或调整实现）进行修正，形成反馈闭环。
上述流程中的每个阶段都以规范文档为中心：代码始终服务于最新的规范。一旦某阶段产出不理想，及时在人干预下修订相应文档，再驱动 AI 继续后续步骤。这种人机协作的迭代，通过在每个阶段加入检查，避免了“小问题酿成大返工”——AI 从一个不变的正确规范出发，每一步都受到约束和校验，从而实现高质量且可预测的产出。最终的 Green Build 表明需求—规范—实现是闭环一致的，规范文档也可作为活文档留存，指导后续维护和迭代。
## 4. 最小可用流程（MVP 实施指南）
即使在不借助 Spec Kit 工具的情况下，团队也可以遵循上述思想，以轻量方式实践规范驱动开发。关键是在项目中建立相应的目录结构和协作流程，模拟 Spec Kit 的产物：
* 目录与文档结构：在项目代码仓库中创建一个specs/目录，作为存放规范和计划的中心。例如，可以按照功能或需求编号建立子目录：specs/001-<功能简述>/spec.md, plan.md, tasks.md 等。对于小型项目或单一功能，也可以直接在 docs/ 或 specs/ 下放置这三个 Markdown 文件。确保所有团队成员都知道这些文件的位置及作用，使其成为单一可信源（single source of truth）。
**人工执行 Specify-Plan-Tasks 流程：**没有 CLI 工具的情况下，可以手动进行每一步：
    1. 撰写 spec.md：收集需求反馈后，产品经理或开发人员使用上述 Prompt 模板（或手工参考 PRD 模板要点）起草规范文档。可借助 ChatGPT 等通用大模型来生成初稿，然后由人修改定稿。务必确保包含用户故事、验收场景（涵盖前端交互与业务逻辑）、功能清单、成功标准等部分。
    2. 撰写 plan.md：将 spec.md 的内容提供给 AI，要求其生成实现方案，或者由技术负责人依据规范撰写计划文档。内容包括架构设计、所需的前后端技术栈、模块划分、接口/data模型定义、第三方依赖等。如有后端接口需求，此时应在 plan.md 中定义接口契约（例如请求路径、参数和响应格式）。人工审核该计划，确保它既符合规范意图又符合现有系统约束。
    3. *撰写 tasks.md：**以 spec.md 和 plan.md 为依据，人工将工作分解成小任务列表，或让 AI 协助生成任务清单。每个任务描述应明确具体工作，例如“更新某页面UI元素”或“实现API方法 X 返回正确数据”。如果使用 AI，可提示它按照每个用户故事拆解任务，并标注任务的先后依赖。整理出的 tasks.md 列表可以按照优先级或阶段分组，并由人调整以贴合团队实际（删掉不必要的，补充遗漏的）。
* 逐步实现与测试：开发团队根据 tasks.md 开始编码实现。推荐的做法是以任务为单位提交代码：每完成一个任务就进行一次代码提交和基本测试。可以使用 AI 编码助手（如 GitHub Copilot 在编辑器中）来加速实现，每次实现前将相关的规范内容提供给 AI作为参考上下文，让它“知道”当前任务对应的需求背景。例如，在实现某接口时，将 spec.md 中该接口相关的验收场景和 plan.md 中的接口契约片段提供给 AI，让其按照规范产出代码。每完成一项任务后，运行对应的测试或进行必要的手工验证，确保验收场景通过。这种做法类似于测试驱动开发（TDD），以验收标准来验证每个小功能点。
* 人工检查与反馈循环：由于缺少自动化的检查工具，团队需要更加勤勉地手工把关。在每个阶段结束时，指定负责人评审文档：Spec是否清楚完整、无含糊？Plan是否合理可行？Tasks是否覆盖所有需求？在编码过程中，也定期核对规范与实现的一致性，发现偏差及时纠正（必要时更新 spec.md 并通知相关开发人员）。通过代码评审和测试，确认最终实现满足 spec.md 中列出的所有验收条件，再进行合并发布。若在测试或评审中发现遗漏的需求或新的边界情况，返回修改规范文档和任务列表，进入下一轮实现，直到测试全部通过。
* 文件管理和协同：将 spec.md、plan.md、tasks.md 纳入版本控制（Git）并与代码变更关联。当某个需求交付完成时，可在版本库中保留这些文档，与代码一起演化（即spec-anchored的做法）。例如，在 Pull Request 中引用相关的规范段落，或在 commit 信息中关联任务编号，方便审查者了解**“为何改动”**。在后续迭代中，若需求变化，则先修改相应 spec.md，再按照流程更新实现，这样规范文档始终反映当前系统行为。
通过上述最小流程，即使不依赖任何专用工具，团队也能践行 Spec-Driven 的原则：让规范先行，再驱动实现。在这个过程中，Markdown 文档和人工介入相结合，可以较好地控制 AI 编码的方向。前端开发人员可据此明晰交互细节，后端开发人员理解业务逻辑接口，避免各自猜测。最终效果是，尽管过程稍有严格，但换来更少的反复调试和更高的一次成功率。当团队养成这种规范驱动的协作习惯后，即便将来引入 Spec Kit 等工具，也不过是水到渠成地自动化已有流程，从而更进一步提升效率和一致性。
**参考资料：**Spec-Driven 方法由 GitHub 等提出并在社区迅速发展，本指南引用和整合了相关最佳实践等，供工程团队在实际项目中落地执行。
