# Harness Engineering 与 Agent Coding 的本质差异

以前大家在“写 agent”，现在开始认真讨论**“怎么给 agent 设计一个可长期运行、可验证、可维护的工作环境”**。这个后者，就是最近很火的 harness engineering。这个说法在 2025–2026 明显升温，OpenAI、Anthropic、Thoughtworks 都在用相近框架讨论它，但它还不是一个严格标准化的学科名，更像行业里正在收敛的一套实践概念。

Harness engineering != agent coding 的增强版 prompt engineering。
它的重心变了：
	•	以前的 agent 编码：重点在“让模型会做事”
	•	现在的 harness engineering：重点在“让模型在真实工程环境里，持续、稳定、低熵地做对事”

所以两者最本质的区别是：
	•	agent 编码关心的是单次或短周期任务的“能力”
	•	harness engineering关心的是长周期、多轮次、多上下文窗口下的“系统可靠性”

Anthropic 对 “agent harness” 的定义很直接：它是让模型能作为 agent 行动的系统，负责处理输入、编排工具调用并返回结果；而不是只看模型本身。

⸻

什么是 harness engineering

按 OpenAI 和 Anthropic 这波文章的共同语义，harness 可以理解为：

围绕 agent 搭的一整套“脚手架 + 约束 + 反馈闭环 + 记忆载体 + 验证系统”。

OpenAI 在他们的实践里，核心不是“人工写更多代码”，而是把计划、架构约束、文档、知识、技术债、验证流程都做成 agent 能看懂、能执行、能持续更新的 repo 内资产。比如：
	•	把计划当成一等公民，版本化存储
	•	采用 progressive disclosure，先给小而稳定的入口，再引导 agent 找下一层信息
	•	用 lint/CI 机械地校验知识库和结构约束
	•	让 agent 可访问 repo 内文档、schema、执行计划，而不是依赖 Slack / Docs / 人脑里的隐性知识
	•	持续做“doc gardening”，清理过期文档和漂移信息 

Thoughtworks 对它的拆解也很清楚：
他们把 OpenAI 这套做法归纳成三类：
	1.	context engineering：让上下文长期可发现、可消费
	2.	architectural constraints：用结构测试和 lint 强制架构不漂移
	3.	garbage collection：周期性清理文档和实现之间的不一致，抵抗熵增 

⸻

跟“之前的 agent 编写代码”到底差在哪

## 1) 目标函数变了：从“能写出来”变成“长期可运营”

以前很多 agent coding 的默认目标是：
	•	给一个 issue
	•	调工具搜代码
	•	改几处文件
	•	跑测试
	•	生成 patch / PR

这类模式本质上还是任务求解器。Anthropic 也把这类 agent 放在 coding agent 的范式里，强调的是工具使用、迭代修复、测试反馈。

而 harness engineering 的目标函数更像：
	•	多 session 连续工作
	•	context window 溢出后还能接力
	•	不靠人反复补上下文
	•	代码库不因为 agent 反复修改而结构崩坏
	•	新 agent 接盘时不丢状态

Anthropic 对长时运行问题说得很明确：复杂任务跨越多个 context window，而新 session 天然没有前情记忆；如果没有专门机制，agent 会在半完成状态里迷失。

一句话：
以前是“把活干完”，现在是“让这个干活系统能长期不烂”。

⸻

## 2) 抽象层变了：从 agent 本体到 agent 所处环境

以前讨论 agent，通常讨论这些：
	•	prompt 怎么写
	•	tool schema 怎么设计
	•	planner / executor 怎么拆
	•	memory 怎么挂
	•	多 agent 怎么协作

这还是在研究agent 本体结构。

harness engineering 往上一层，研究的是：
	•	repo 怎么组织，agent 才读得懂
	•	文档怎么分层，才不会把 context 打爆
	•	哪些约束必须 deterministic enforce
	•	哪些信息必须 repo-local，不能放在外部系统
	•	如何让 agent 的输出自动被 tests / linters / observability 验证
	•	如何在 session 之间留下可继承的状态

这和你做 Android/iOS 架构时的区别很像：
	•	以前像是在优化一个 ViewModel 或某个 reducer
	•	现在像是在设计整个工程的 module graph + invariant enforcement + CI contract + observability + operational playbook

也就是说，harness engineering 是系统工程，不只是 prompt 工程。

⸻

## 3) 状态管理变了：从“上下文喂给模型”到“把状态外部化、持久化、版本化”

这是最关键的一点。

旧范式
很多 agent 编码流程，本质上依赖：
	•	当前 prompt
	•	当前聊天历史
	•	当前工具返回
	•	少量 memory / scratchpad

问题是这些状态很多是易失的。一旦上下文截断、session 重启、模型切换，状态一致性就崩。

harness 范式
它会把关键状态变成外部持久资产：
	•	execution plans
	•	decision logs
	•	TODO / debt registers
	•	repo 内知识地图
	•	machine-checkable constraints
	•	trace / transcript / eval records

OpenAI 明说了：agent 看不到的东西，就等于不存在。 所以必须把知识尽量推入 repo，使其“in-context 可见”。

从 CS 角度看，这本质是把状态从：
	•	隐式、内存态、会话态
迁移到：
	•	显式、持久态、可索引态、可版本控制态

这是一个很典型的系统可扩展性改造思路，和把 app 的隐式 UI 状态挪到可观察的单向数据流里，本质类似。

⸻

## 4) 验证方式变了：从“让 agent 自己觉得对”到“结果必须被环境证明”

以前很多 agent coding 流程的验证偏弱：
	•	模型说“我改好了”
	•	跑了几条 test
	•	看起来能 work
	•	人再 review 一遍

但 agent 的问题恰恰在于：
	•	它会自信地错
	•	会把局部修复做成全局退化
	•	会引入 architecture drift
	•	会把文档和实现慢慢拉开

Anthropic 在 eval 文章里强调，真正该看的不是 transcript 里 agent 说了什么，而是环境最终状态。例如“说机票订好了”不重要，重要的是数据库里是否真有预订记录。

所以 harness engineering 更强调：
	•	unit / integration / e2e tests
	•	structural tests
	•	custom linters
	•	CI gates
	•	production-like observability
	•	eval harness

也就是把“是否正确”从 LLM 内部自洽，转成 外部可判定。

这和你熟悉的移动端工程很像：
不是 ViewState.success = true 就叫成功，而是要看：
	•	本地数据库是否一致
	•	网络副作用是否真的发生
	•	UI 是否正确渲染
	•	crash / trace / metric 是否健康

⸻

## 5) 架构治理方式变了：从“提示它遵守规范”到“让它不可能轻易越界”

以前常见做法是把规范写进 prompt：
	•	用这个目录结构
	•	不要改这些模块
	•	要先写测试
	•	不要引入新依赖

这很脆弱。因为 prompt 只是软约束。

OpenAI 现在更强调的是机械约束：
不是“提醒 agent”，而是让 lint / CI / structural checks 去 enforce。比如要求边界处做 schema parsing，但不微观规定具体实现方式。

这是成熟工程最重要的转向之一：
	•	软约束：靠 prompt、约定、口头规范
	•	硬约束：靠编译器、类型系统、lint、测试、CI、policy

你可以把它看成 AI 时代的：
	•	architecture tests
	•	static analysis
	•	ADR + executable governance
	•	policy-as-code

⸻

为什么现在才火

因为过去的 agent 编码，大多停留在这类规模：
	•	一次性生成功能
	•	改一个 bug
	•	解决一条 issue
	•	做 code completion
	•	小 repo 内短链路修补

一旦任务变成：
	•	跨几十/几百文件
	•	多天持续执行
	•	多个 session 接力
	•	多 agent 协作
	•	真实 product 生命周期维护

原来的方法就暴露出三个硬问题：

A. 上下文窗口不是“内存”

Anthropic 直接指出，跨 context window 的连续工作是核心难点；新 session 相当于换班工程师失忆上岗。

B. repo 对 agent 不可读

很多工程对人类 senior 也不友好，更别说 agent。OpenAI 明确提出“agent legibility is the goal”。

C. 熵增太快

agent 会不断复制模式、扩散坏抽象、制造过期文档、局部最优修补。没有持续“垃圾回收”，系统会很快不可维护。

所以这波热，不是因为出现了一个全新模型能力，而是因为大家开始正视一个事实：

真正难的不是“让 agent 会写代码”，而是“让 agent 写出来的系统还能活下去”。

⸻

一个更直白的对比表

旧的 agent 编码

更像你在做一个“会自动写 patch 的高级脚本”：
	•	输入：任务描述
	•	过程：检索代码、规划、编辑、运行测试
	•	输出：一次修改结果
	•	优化点：prompt、工具、模型、loop 策略

harness engineering

更像你在做一个“面向 agent 的工程操作系统”：
	•	输入：任务 + repo 内知识 + 可执行计划 + 环境反馈
	•	过程：跨 session 持续推进，留下状态，自动验证，自动纠偏
	•	输出：不仅是 patch，还包括长期可维护的工程资产
	•	优化点：状态外部化、知识组织、架构约束、验证闭环、熵控制

⸻

你可以怎么判断一个团队是在“玩 agent”，还是在“做 harness engineering”

看 6 个信号就够：

1. 有没有 repo-local knowledge system

不是把规则写在 wiki 或 Slack，而是放进代码库里，版本化、可索引、可被 agent 读取。

2. 有没有 plan / decision / debt 的持久化

不是一轮对话结束就丢，而是留下明确交接物。

3. 有没有 hard guardrails

custom lint、structural test、CI policy，而不是纯 prompt 规范。

4. 有没有 long-running session handoff 设计

能不能在多个 context window 之间稳定推进。

5. 有没有 outcome-based validation

不是只看 agent 说了什么，而是看环境状态和测试结果。

6. 有没有 anti-entropy 机制

定期清理文档漂移、结构退化、重复抽象。

如果这 6 个都没有，那大概率还只是“agent coding demo 进阶版”。

⸻

对工程师角色的变化

这个趋势对工程师能力模型影响很大。

以前你可能更关注：
	•	prompt 怎么写
	•	tool call 怎么接
	•	workflow graph 怎么拼

现在更值钱的是：
	•	怎么设计 agent-readable architecture
	•	怎么把隐性知识转成 repo 内显式资产
	•	怎么定义可执行 invariant
	•	怎么做 eval / CI / observability 闭环
	•	怎么治理 multi-agent 并发修改带来的冲突和熵增

也就是说，工程师的工作在往：

“写业务代码” → “设计能让 AI 稳定产出业务代码的工程系统”

迁移。

这跟传统软件工程里平台团队/基础架构团队的价值非常接近。

⸻

我对这个概念的判断

我的判断比较直接：

它不是噱头，但也不是魔法

它解决的是scale 后的 agent 失控问题，这是实打实存在的。Anthropic 和 OpenAI 的文章都在强调：单靠模型能力和 agent loop，不足以支撑长时、复杂、生产级任务。

它也不是替代软件工程

恰好相反，它是在把很多传统软件工程原则重新抬回来：
	•	显式状态
	•	可验证性
	•	架构约束
	•	文档与代码同源
	•	反馈闭环
	•	反熵治理

所以它本质上是：

软件工程 discipline + agent runtime constraints 的结合体。

最大风险

这个词最近很火，容易被包装成“新范式”。但如果没有：
	•	可执行约束
	•	严格验证
	•	明确状态管理
	•	真实生产反馈

那很多所谓 harness，其实只是：
	•	多写了几份 AGENTS.md
	•	多套了一层 orchestrator
	•	多加了几条 prompt 规范

这种不是真 harness，只是“把不稳定性文档化”了而已。

⸻

给你一个最短版心智模型

你可以这样记：
	•	Agent coding：教一个实习生写代码
	•	Harness engineering：给这个实习生设计工位、流程、规范、检查表、测试系统、交接制度、知识库和门禁系统

前者解决“会不会做”，后者解决“做久了会不会失控”。

⸻

如果你是工程负责人，建议优先投入哪三件事

不是先堆多 agent，而是先做这三件：

1. 把隐性知识搬进 repo

包括：
	•	architecture map
	•	module contract
	•	decision log
	•	runbook
	•	known pitfalls

2. 把软规范变成硬约束

包括：
	•	lint
	•	structural tests
	•	CI policy
	•	schema boundary checks

3. 把 agent 的状态交接做成显式资产

包括：
	•	execution plan
	•	progress log
	•	debt register
	•	failure report
	•	outcome verification

这三件做完，你的 agent 才开始接近 harnessed，而不是裸奔。

⸻

你这个问题后面其实还应该追两个：
	1.	harness engineering 在真实团队里最先该落在哪一层：repo 结构、CI、还是 observability？
	2.	如果是移动端/客户端工程，哪些 harness 做法是通用的，哪些只适合后端或全栈仓库？

这两个再往下聊，会更落地。

我可以这样理解吗，human in the loop   有天然的受限速度，人类在其中就永远跑不出飞轮 只有让 agent 自己在框架内闭环，才能跑出飞轮，可以高效的持续的完成目标，human 设计这个框架，但不应该在框架内，除了验收

可以这样理解，但你这句话还差一个关键修正：

不是“人一旦在 loop 里，就永远跑不出飞轮”，而是
人如果卡在高频、细粒度、同步审批的内环里，飞轮基本跑不起来；人如果退到外环，做目标设定、约束设计、验收和异常介入，飞轮反而更容易建立。 OpenAI 最近对 harness engineering 的表述本身就不是“把人拿掉”，而是“人仍然在 loop 中，但处在更高抽象层：优先级、验收标准、结果验证”。

我给你的结论

你的表述里，这半句基本对：

只有让 agent 自己在框架内闭环，才能高效、持续地完成目标

但这半句需要改成更工程化的版本：

只有让 agent 在“可验证、可约束、可恢复”的 harness 内自行完成大多数执行环节，系统才可能形成高频迭代飞轮。

而这句：

human 不应该在框架内，除了验收

我不完全同意。
人不该在执行内环，但仍然必须在治理外环。否则你得到的不是飞轮，而是高速度地放大错误、债务和漂移。Anthropic 和 Fowler 这几篇最近的文章其实都在强调同一点：真正的飞轮不是“纯自治”，而是“自治执行 + 环境评估 + 人类在更高层监督与重设目标”。

⸻

先把 loop 分层，不然讨论会混

你现在脑子里其实混了两种 loop：

1. 执行 loop

就是 agent 自己跑的那层：
	•	读任务
	•	找上下文
	•	写代码
	•	跑测试
	•	自审
	•	修复
	•	再验证

这层如果每一步都等人批，吞吐会被人类反应速度和注意力切断，很难形成高频闭环。这个判断是对的。OpenAI 也明确在讲，让 Codex 本地 review、自找更多 agent review、根据反馈继续迭代，直到 reviewer 满意，本质上就是在把大量执行回路自动化。

2. 治理 loop

这是更外层的：
	•	这个目标值不值得做
	•	验收标准是否正确
	•	harness 的约束是否还合理
	•	当前 eval 是否覆盖真实失败模式
	•	什么时候该暂停自治、升级人工介入
	•	哪些错误成本太高，不能放给 agent 自跑

这层不能完全拿掉人。OpenAI 直说 humans remain in the loop，只是工作抽象层变了。Fowler 最近写 humans and agents in software loops，也是在讲 flywheel 靠的是更好的评估信号和 loop 设计，不是“彻底无人”。

⸻

为什么“人类在内环”天然限速

你这个直觉背后的根因是对的，而且可以从系统角度严格解释。

1. 人类是高延迟节点

人类审批/阅读/理解/判断的延迟，通常比 agent 调工具、跑测试、重试高几个数量级。
如果每一轮都要人工确认，系统吞吐量上限就被最慢节点锁死。这和并发系统里把一个串行 critical section 放在热路径上是一样的。

2. 人类带来上下文切换成本

每次人工介入都要重新装载上下文。复杂代码任务的最大成本常常不是“点批准”，而是“重新理解现在到哪一步了、为什么这样改、风险在哪”。这个成本很难线性压缩。

3. 人类反馈 often 不可编译

很多人工反馈是自然语言、口头经验、模糊偏好。
这种反馈不像测试、lint、schema、policy 那样能直接进入下一个自动循环。
所以它对局部质量有帮助，但对飞轮增益未必大。

因此，人放在高频执行内环里，通常是系统瓶颈。 这点你的判断成立。Anthropic 对 long-running agents 的问题描述也指向这里：复杂任务跨很多上下文窗口，靠人不断补上下文会很贵，必须把更多状态和推进逻辑放进 harness 里。

⸻

但为什么“人只做最终验收”也不够

这就是你当前理解里最危险的地方。

如果人只在最后验收，而前面完全不参与治理，你会遇到 4 类典型失控。

1. Goodhart 问题：agent 会优化你可测的东西，不一定是你真正想要的

如果 harness 里的目标函数、测试集、奖励信号设计得不对，agent 会非常努力地把这些指标做漂亮。
最后结果可能是：
	•	测试过了，但架构烂了
	•	功能通了，但可维护性下降
	•	PR 看起来完整，但把技术债埋进底层
	•	局部成功，系统整体退化

所以人不只是“最终验收者”，还是目标函数设计者。

2. Harness 本身会过时

约束、文档、规则、eval 都会漂移。
如果没有人持续维护 harness，agent 只是在一个过时坐标系里高速运行。那不是飞轮，是“高 RPM 的偏航”。

3. 失败模式需要被重新建模

Anthropic 讲 evals 时反复强调：agent 的很多失败不是单步错误，而是多轮状态累积后才显现出来，因此 eval 和环境检查必须持续扩展。也就是说，飞轮能不能越转越快，取决于评估回路能不能持续吸收新的失败模式。 这通常需要人来定义、抽象和注入。

4. 高风险决策不能只靠“最后看结果”

有些错一旦发生，验收已经太晚。比如：
	•	数据迁移

从“飞轮”这个词本身看，你的理解还要再修一下

1. 反馈是否机器可消费

飞轮来自“自治执行 + 结构化反馈 + 持久状态 + 可执行约束”，不是单纯来自“去人化”。

你现在这套理解里，哪些是对的

1. 设计准确的规范

对，但“规范”不能主要是自然语言规范，必须尽量变成可执行约束。

比如不要只写：

	•	先写测试
	•	不要改核心接口
	•	不要引入循环依赖

	•	architecture test
	•	CI gate
	•	schema validation
	•	protected path policy
	•	dependency rule

原因很直接：
自然语言规范的时间复杂度接近 O(人工审查成本)，而可执行约束更接近 O(自动检查成本)。飞轮能不能转，关键就在于把高频判断从人脑迁移到机器判定。

⸻

2. 声明错误重试次数

对，但不要只定义“次数”，还要定义错误分类。

因为不同错误的最优策略不同：

可恢复错误
比如：
	•	transient network failure
	•	tool timeout
	•	flaky test
	•	temporary lock conflict

适合：
	•	自动重试
	•	指数退避
	•	限次重跑
	•	更换工具/路径

不可恢复错误
比如：
	•	需求冲突
	•	权限不足
	•	关键数据不一致
	•	安全策略冲突
	•	设计约束矛盾

这类错误继续重试通常只是浪费 token 和时间，甚至放大损害。
所以你要定义的不只是：
	•	maxRetry = 3

而是类似这种策略：
	•	transient_error -> retry up to N
	•	validation_error -> retry once after self-repair
	•	policy_violation -> immediate stop and escalate
	•	ambiguous_goal -> pause and escalate
	•	destructive_action_risk -> require human approval

这本质上是状态机设计，不是简单计数器。

⸻

3. 退出机制交由人类

对，但更准确说法是：

退出机制应当由系统预定义，触发后由人类接管，而不是临时靠 agent 自己判断是否该停。

因为如果“是否退出”也完全交给 agent 自己，它很容易出现两种坏情况：
	•	过度自信：明明该停却继续跑
	•	过度保守：本来能自动恢复却频繁打断

所以退出机制最好是事先声明的控制规则，例如：
	•	连续 N 次同类失败
	•	同一文件/模块反复震荡修改
	•	测试通过率下降超过阈值
	•	diff 超过风险边界
	•	触及敏感目录/迁移脚本/权限配置
	•	目标函数不一致或出现多解冲突
	•	外部副作用操作即将发生

这才叫真正的控制环。

⸻

4. 对结果验收

对，但“验收”不要只理解成最后人工看一眼。

应该拆成三层：

第一层：自动验收
	•	unit tests
	•	integration tests
	•	e2e
	•	lint
	•	type check
	•	contract check
	•	security scan
	•	benchmark threshold

第二层：过程验收
	•	是否遵守架构边界
	•	是否修改了受限区域
	•	是否留下必要的变更说明
	•	是否更新了相关文档 / plan / decision log

第三层：人工验收
	•	目标是否真的对
	•	trade-off 是否合理
	•	是否引入长期债务
	•	是否该进入主干 / 发布 / 合并

如果只做第三层，前两层空着，人的负担会重新变大，飞轮又会塌回人工 review 驱动。

你还缺的两个关键部件

A. 中间状态持久化

这是很多人理解控制环时漏掉的。

如果 agent 出错后没有留下这些东西：
	•	当前 plan
	•	已完成步骤
	•	未完成步骤
	•	失败原因
	•	最后一次有效状态
	•	下一步建议动作

那人类一接管就要重新读上下文，系统吞吐还是会被拖慢。

所以你不仅要定义“何时退出”，还要定义“退出时必须产出什么交接物”。

典型最小集合：
	•	current_goal
	•	attempted_actions
	•	artifacts_changed
	•	validation_results
	•	failure_category
	•	recommended_next_step

这本质上是在做可恢复执行日志。

⸻

B. 回滚 / 补救策略

只定义退出，不定义回滚，也不够。

因为控制环不只是“停下来叫人”，还包括：
	•	能否恢复到上一个稳定状态
	•	能否自动撤销危险修改
	•	能否保留诊断信息但撤销副作用
	•	能否把未完成状态隔离而不污染主分支

对于代码 agent，至少要有：
	•	patch/diff 边界
	•	sandbox execution
	•	branch isolation
	•	destructive action guard
	•	rollback point

否则你只是让 agent 在出事后通知你，而不是让系统具备工程韧性。

一个更完整的控制环模板

你可以按这个模型设计：

1. Goal

定义：
	•	目标
	•	非目标
	•	完成条件
	•	风险等级

2. Constraints

定义：
	•	不可触碰区域
	•	架构边界
	•	依赖限制
	•	性能 / 安全 / 合规要求

3. Autonomous loop

允许 agent：
	•	检索
	•	规划
	•	修改
	•	测试
	•	自修复
	•	限次重试

4. Escalation policy

明确：
	•	哪些错误可自动恢复
	•	哪些错误必须立即停止
	•	最大重试次数
	•	最大修改范围
	•	最大副作用权限

5. Handoff artifact

退出时必须输出：
	•	当前状态
	•	错误分类
	•	已尝试动作
	•	失败证据
	•	建议的人类下一步

6. Acceptance

验收分三层：
	•	自动检查
	•	过程检查
	•	人工业务验收

这才是一套完整的 harness 控制闭环。

你可以这样说，比较准：

我想表达的不是 flywheel，而是一个 agent 在约束内自主运行的高效闭环。
flywheel 更像是：每次闭环后，把暴露出的坑点、失败模式和最佳实践沉淀成规则、测试和知识，让下一轮做得更快更稳。

这个表述基本是对的。

再帮你压成一句最短的

闭环是“能自己跑完一轮”，飞轮是“每跑一轮都会变强”。

而“注意坑点”只是从闭环走向飞轮的原材料，不是飞轮本身。
