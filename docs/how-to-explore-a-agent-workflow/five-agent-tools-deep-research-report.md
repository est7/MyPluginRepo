# Agent Workflow 仓库深度研究方法论

## 分析框架建议

这套 workflow 的失效边界、代价模型，以及哪些东西值得抽象复用。

只写"怎么用"和"哪里亮眼"还不够，因为那仍然偏展示层。真正能把 10 个仓库挖出价值差异的，是把它们拉回到工程基本面：

- 它解决的根问题到底是什么
- 它通过什么机制解决
- 为此付出了什么代价
- 在哪些场景下会失效
- 哪些设计可以迁移，哪些只是绑定特定宿主/模型/团队文化

这个第三部分会让你的分析从"读后感"变成"可复用的方法论"。

---

## 每个仓库按 6 个维度拆

这样最后 10 个仓库可以横向对比，不会散。

### 1. 使用方式：基础流 + 进阶流

建议拆成：

**基础流**

给第一次接触的人看，回答"最小可用路径是什么"。

格式可以固定成：

- 安装 / 初始化
- 最小 happy path
- 输入物是什么
- 输出物是什么
- 中间状态如何看
- 失败后怎么恢复

**进阶流**

给已经会跑的人看，回答"复杂场景怎么用"。

建议重点写：

- 多阶段任务怎么拆
- 中断恢复怎么做
- 如何处理 verify / review / rollback
- 如何做 handoff / collaboration
- 如何约束 scope
- 如何和外部工具协作（git、tests、CI、issue tracker、docs）

这里不要只是列命令，要写成状态转移。

例如不是写：

- plan
- apply
- unify

而是写：

- 从"需求模糊"进入"有 acceptance criteria 的计划"
- 从"计划已确认"进入"执行 + 复核循环"
- 从"已完成实现"进入"结果对账 + 状态提交"

这样你在看不同仓库时，才能比较它们是不是同一个抽象，只是名字不同。

---

### 2. 亮点设计：具体到文件、片段、机制

不要只写"这个设计很妙"，而要固定回答 5 个问题：

- 在哪个文件
- 原文或核心片段是什么
- 它试图防什么失败模式
- 它是软约束还是硬约束
- 为什么这比常见做法更好

例如：

- `src/workflows/apply-phase.md`
- 引用 E/Q loop 的原文
- 目标失败模式：自我乐观、未重新读取产物
- 约束类型：软约束（prompt discipline）
- 优势：把"执行完成"提升为"验证通过才算完成"

这样你的"亮点"会更像设计审查，而不是摘抄。

---

### 3. 失效边界 / 代价模型 / 可复用抽象

这是最建议加的第三部分。

每个仓库都写这三块：

**A. 失效边界**

它在哪些条件下容易不 work。

例如：

- 强依赖特定模型的服从性
- 依赖单线程工作方式，多人并行就混乱
- 依赖 markdown state，分支冲突难处理
- 小任务体验好，大任务上下文会爆
- 对代码仓友好，对产品/运营流程不友好
- 要求用户 discipline 很高，否则很快绕开系统

这部分很重要，因为"亮点"没有边界条件就容易被高估。

**B. 代价模型**

它为了稳定性、可审计性、可恢复性，付出了什么成本。

建议固定看这些成本：

- ceremony 成本：步骤多不多
- token 成本：上下文长不长
- latency 成本：要不要多轮 verify / reread
- 认知成本：用户要学多少概念
- 文件维护成本：state / summary / roadmap 会不会膨胀
- 集成成本：接入新工具难不难
- 误判成本：路由错了会怎样

这是区分"看起来先进"和"实际可用"的关键。

**C. 可复用抽象**

最重要的一问：

如果不整套搬，只抽 1–3 个机制，哪些值得复用？

例如一个仓库你最后可能得出：

- 值得复用：任务状态机、verify before success、phase closure
- 不值得照搬：大量 ceremony、宿主绑定命令、冗长模板

这样 10 个仓库最后你能沉淀出自己的 workflow design kit。

## 完整扩展框架：五段式结构

最终写成五段式，不是三段式。

### 第一段：使用说明

回答"怎么跑起来"。

内容：

- 基础流
- 进阶流
- 常见命令 / 入口
- 文件结构总览
- 最小案例

这是使用者视角。

---

### 第二段：设计亮点

回答"为什么这么设计"。

内容：

- 核心设计决策
- 具体文件与原文引用
- 对应失败模式
- 机制如何运作
- 与常见方案对比

这是设计者视角。

---

### 第三段：失效边界与代价

回答"它什么时候不该用，为什么不便宜"。

内容：

- 适用场景
- 不适用场景
- 成本与复杂度
- 宿主依赖
- 并发/团队协作限制
- 长周期维护问题

这是架构评估视角。

---

### 第四段：可迁移模式

回答"哪些东西可以抽象成通用模式"。

这是最容易产生你自己方法论的部分。

建议给每个仓库提炼：

- **State pattern**：状态文件、phase、checkpoints
- **Loop pattern**：plan/apply/review/unify 之类闭环
- **Verification pattern**：何时验证，验证什么，谁负责
- **Recovery pattern**：失败后如何回滚/重规划
- **Scope control pattern**：如何避免漂移
- **Memory pattern**：如何跨 session 持续上下文
- **Dependency gating pattern**：外部工具、skills、permissions 怎么阻断
- **Ceremony compression pattern**：小任务如何降流程成本

你会发现很多仓库表面不同，但底层其实是在这些 pattern 上组合。

---

### 第五段：你的判断与重构建议

回答"如果你来重做，会怎么改"。

这个部分最有价值，因为它能把"挖掘"升级成"批判性吸收"。

每个仓库都可以写：

- 我认可的核心判断是什么
- 我不认可的前提是什么
- 如果面向 Android / backend / fullstack 团队，我会怎么改
- 哪些软规则应该变硬门禁
- 哪些状态应该落文件，哪些应该落工具
- 哪些流程应该自动化，哪些必须保留人为判断

这样最后不是"10 篇仓库阅读笔记"，而是"1 套你自己的 workflow 设计原则"。

## 统一评估表

每个仓库都打同一套标签，不然最后没法归纳。

### 基本信息

- 仓库名
- 目标用户
- 宿主环境（Claude Code / Cursor / shell / GitHub Actions / custom agent）
- 状态持久化方式（markdown/json/yaml/db）
- 核心入口（commands/hooks/scripts/templates）

### 工作流结构

- 是否有显式 phase
- 是否有状态机
- 是否有 verify gate
- 是否有 failure routing
- 是否有 handoff / resume
- 是否有 closure / reconcile
- 是否支持 quick path / compressed ceremony

### 成本与风险

- 上手成本
- 日常使用摩擦
- token 成本
- 多人协作适配度
- 并发适配度
- 对模型能力依赖程度
- 对用户 discipline 依赖程度

### 结论

- 最值得借鉴的 3 点
- 最不建议照搬的 3 点
- 适用场景
- 不适用场景
- 推荐指数

这样你后面可以做真正的横向分析，而不是只看单仓。

## 反模式章节

这类 workflow 仓库会反复踩同样的坑：

- **把格式化当状态管理**：有很多模板，不代表有真正状态机
- **把命令数量当能力**：命令多，可能只是表层 API 多，底层抽象并没增强
- **把 summary 当归档，不当 reconciliation**：没有 closure 的总结，通常只是日志
- **把 verify 放到最后**：这会导致错误扩散，而不是局部收敛
- **把所有失败都当 code issue**：实际很多失败是 intent/spec 问题
- **把 markdown 文件当万能数据库**：审计好，但并发和分支管理可能很差
- **小任务 ceremony 过重**：用户最终一定绕开系统
- **完全依赖 prompt 纪律**：软约束无法替代硬门禁

这个"反模式库"最后会比单个仓库分析更通用。

## 每个仓库的落地模板

你之后每个仓库都可以按这个模板写：

**1. 基础使用**

- 安装与初始化
- 最小 happy path
- 输入 / 输出 / 状态文件
- 常用命令
- 常见错误与恢复

**2. 进阶使用**

- 复杂任务拆分
- 中断恢复
- verify / handoff / rollback
- 与外部工具协作
- 团队协作方式

**3. 核心设计与亮点决策**

- 关键文件
- 原文引用
- 机制解释
- 试图解决的失败模式
- 为什么这比常见做法更强

**4. 失效边界与代价模型**

- 适用场景
- 不适用场景
- ceremony / token / cognitive cost
- 并发与团队协作问题
- 对模型与用户的依赖

**5. 可复用模式与我的结论**

- 值得迁移的 3 个模式
- 不建议照搬的 3 个设计
- 如果我来重写会怎么改
- 适合什么团队 / 项目类型

## 建议的两个扩展

### 扩展一：模式词典

不要只写仓库分析，顺手沉淀术语表。例如：

- closure
- reconciliation
- state continuity
- failure routing
- ceremony compression
- scope guard
- verification gate
- anti-rationalization
- dependency gating
- session resume

每个术语都给：

- 定义
- 解决什么问题
- 常见实现方式
- 哪些仓库用了
- 你的评价

最后你会得到一套可复用语言，这对你以后自己设计 workflow 非常重要。

### 扩展二：自己的 workflow blueprint

等你分析到第 5–6 个仓库时，就不要只被动读了，开始同步写：

- 我的最小状态机
- 我的 verify gate
- 我的 failure routing
- 我的 state persistence
- 我的 quick path
- 我的 team mode

这样等 10 个读完，你不是得到 10 份笔记，而是已经有一版自己的 workflow draft。

## 最终研究目标

更高价值的目标应该是：

从十个仓库里抽出一套面向工程团队、可验证、低 ceremony、支持恢复与协作的 workflow 设计原则。

这才是研究的复利所在。
