# 长运行应用开发的 Harness 设计

**发布于 2026 年 3 月 24 日**

Harness 设计是代理编码前沿性能的关键。下面是我们如何在前端设计和长运行自主软件工程中进一步推动 Claude 的发展。

*由我们的 Labs 团队成员 Prithvi Rajasekaran 撰写。*

过去几个月，我一直在研究两个相互关联的问题：让 Claude 生成高质量的前端设计，以及让它在没有人工干预的情况下构建完整的应用程序。这项工作起源于我们早先在 [前端设计技能](https://github.com/anthropics/claude-code/blob/main/plugins/frontend-design/skills/frontend-design/SKILL.md) 和 [长运行编码代理 Harness](https://www.anthropic.com/engineering/effective-harnesses-for-long-running-agents) 上的努力，我的同事和我通过提示工程和 Harness 设计将 Claude 的性能提升到远高于基线水平——但两者最终都碰到了天花板。

为了突破瓶颈，我寻找了一种能在两个截然不同的领域都适用的新型 AI 工程方法：一个领域由主观品味定义，另一个由可验证的正确性和可用性定义。从 [生成对抗网络](https://en.wikipedia.org/wiki/Generative_adversarial_network)（GANs）中获得灵感，我设计了一个多智能体结构，包含 **生成器** 和 **评估器** 两个智能体。要构建一个能可靠且有品味地评分输出的评估器，首先需要制定一套标准，将“这个设计好吗？”这类主观判断转化为具体、可评分的术语。

随后，我将这些技术应用到长运行自主编码中，继承了早期 Harness 工作的两个经验：将构建分解为易于处理的块，以及使用结构化工件在会话之间传递上下文。最终成果是一个三智能体架构——规划器、生成器和评估器——能够在多小时的自主编码会话中产出丰富的全栈应用程序。

## 为什么朴素实现不够用

我们此前已证明，Harness 设计对长运行代理编码的有效性有重大影响。在早期的 [实验](https://www.anthropic.com/engineering/effective-harnesses-for-long-running-agents) 中，我们用初始化智能体将产品规格分解为任务列表，再由编码智能体逐个功能实现，最后通过工件在会话间传递上下文。更广泛的开发者社区也得出了类似洞见，例如使用钩子或脚本让智能体保持连续迭代循环的 “[Ralph Wiggum](https://ghuntley.com/ralph/)” 方法。

但一些问题依然顽固存在。对于更复杂的任务，智能体随着时间推移仍容易脱轨。在拆解这个问题时，我们观察到执行这类任务的智能体存在两种常见失败模式。

第一种是模型在长任务中随着上下文窗口填满而失去连贯性（详见我们关于 [上下文工程](https://www.anthropic.com/engineering/effective-context-engineering-for-ai-agents) 的文章）。部分模型还会出现“上下文焦虑”，即在接近它们认为的上下文限制时过早结束工作。上下文重置——彻底清空上下文窗口并启动全新智能体，同时通过结构化交接携带前一个智能体的状态和后续步骤——能同时解决这两个问题。

这与“压缩”不同，压缩是在原地总结对话早期内容，让同一智能体在缩短的历史上继续工作。虽然压缩保留了连续性，但它没有给智能体一张“干净的石板”，上下文焦虑仍可能存在。重置提供干净石板，但代价是交接工件必须携带足够的状态，让下一个智能体能无缝接手。在早期测试中，我们发现 Claude Sonnet 4.5 的上下文焦虑非常严重，仅靠压缩不足以实现强大的长任务性能，因此上下文重置成为 Harness 设计的核心要素。这解决了根本问题，但也为每次 Harness 运行增加了编排复杂度、令牌开销和延迟。

第二个我们此前未解决的问题是自我评估。当被要求评估自己产出的工作时，智能体往往会自信地大加赞赏——即使在人类观察者看来质量明显平庸。这种问题在主观任务（如设计）中尤为突出，因为没有类似于可验证软件测试的二元检查。“布局是否精致还是通用”是一个主观判断，智能体在自我评分时总是偏向正面。

即使在有可验证结果的任务上，智能体有时也会表现出判断力不足，从而阻碍任务完成。将“干活的智能体”和“评分的智能体”分开，被证明是解决这一问题的强有力杠杆。这种分离本身并不能立即消除宽容倾向（评估器仍是 LLM，仍倾向于对 LLM 生成的内容宽容），但调优一个独立的评估器使其保持怀疑态度，要比让生成器批判自己的工作容易得多。一旦有了外部反馈，生成器就有了可以迭代的具体依据。

## 前端设计：让主观质量变得可评分

我先从前端设计入手实验，因为自我评估问题在这里最明显。在没有任何干预的情况下，Claude 通常会倾向于安全、可预测的布局——技术上功能完备，但视觉上平淡无奇。

塑造我为前端设计构建的 Harness 的两个核心洞见是：第一，虽然美学无法完全简化为分数（个人品味永远存在差异），但可以通过编码设计原则和偏好的评分标准来显著提升；“这个设计漂亮吗？”很难一致回答，但“这个设计是否符合我们的好设计原则？”则给了 Claude 具体可评分的依据。第二，将前端生成与前端评分分离，能形成一个反馈循环，推动生成器产出更优秀的输出。

基于此，我为生成器和评估器智能体的提示编写了四条评分标准：

- **设计质量**：设计是否感觉像一个连贯的整体，而不是零散部分的拼凑？优秀的设计应让颜色、排版、布局、图像等细节共同营造出独特的情感和身份。
- **原创性**：是否有自定义决策的证据，还是只是模板布局、库默认值和 AI 生成的模式？人类设计师应能看出刻意的创造性选择。未修改的库存组件——或 AI 生成的典型痕迹（如白色卡片上的紫色渐变）——在此项会失败。
- **工艺**：技术执行，包括排版层级、间距一致性、颜色和谐、对比度比率。这是能力检查而非创造力检查。大多数合理的实现默认就能达标；失败意味着基础不牢。
- **功能性**：独立于美学的可用性。用户能否理解界面功能、找到主要操作，并无需猜测即可完成任务？

我特别强调设计质量和原创性，而非工艺与功能性。Claude 默认在工艺和功能性上得分已较高，因为所需的技术能力对模型来说较为自然。但在设计和原创性上，Claude 往往产出最多平淡无奇的结果。标准明确惩罚高度通用的“AI 渣”模式，通过加重设计与原创性的权重，推动模型敢于进行更多审美冒险。

我使用带有详细分数分解的 few-shot 示例来校准评估器，确保其判断与我的偏好一致，并减少迭代中的分数漂移。

我基于 [Claude Agent SDK](https://platform.claude.com/docs/en/agent-sdk/overview) 构建了这个循环，编排保持简洁。生成器智能体首先根据用户提示创建 HTML/CSS/JS 前端。我为评估器提供了 Playwright MCP，让它在评分前能直接与实时页面交互。在实际运行中，评估器会自主导航页面、截图并仔细研究实现，然后给出评估。该反馈会作为下一个迭代的输入回传给生成器。每个生成过程我运行 5 到 15 次迭代，每次迭代通常会让生成器朝着更独特的方向推进，因为它必须回应评估器的批判。由于评估器是主动导航页面而非仅评分静态截图，每个循环都需要真实的时间。全程运行最长可达四小时。我还指示生成器在每次评估后做出战略决策：如果分数趋势良好就细化当前方向，否则就完全转向另一种审美风格。

在多次运行中，评估器的评分随迭代逐步提升，然后趋于平稳，但仍有提升空间。有些生成是逐步细化，有些则在迭代间发生剧烈的审美转折。

标准的措辞以我未完全预料到的方式引导了生成器。例如包含“最好的设计是博物馆级品质”这样的短语，会让设计向特定视觉风格收敛，表明与标准相关的提示本身就直接塑造了输出的特征。

虽然分数通常随迭代提升，但模式并非总是线性。后期的实现整体更好，但我经常发现自己更喜欢中间某次迭代而非最终版本。随着轮次增加，实现复杂度也倾向于上升，生成器会响应评估器的反馈而追求更具雄心的方案。即使是第一次迭代，输出也明显优于没有任何提示的基线，说明标准和相关语言本身就在任何评估器反馈之前，就已引导模型远离通用默认。

在一个值得注意的例子中，我提示模型创建一个荷兰艺术博物馆网站。到第九次迭代，它产出了一个干净、暗色调的虚构博物馆着陆页，视觉上很精致，但基本符合我的预期。到了第十次循环，它彻底放弃了原有方案，将网站重新构想为一个空间体验：用 CSS 透视渲染的棋盘地板 3D 房间，艺术品以自由位置挂在墙上，通过“门道”而非滚动或点击在画廊房间间导航。这是一种我此前从未在单次生成中见过的创造性飞跃。

## 扩展到全栈编码

掌握这些发现后，我将这种 GAN 启发的模式应用到全栈开发。生成器-评估器循环自然映射到软件开发生命周期——代码审查和 QA 扮演了与设计评估器相同的结构角色。

### 架构

在我们早期的 [长运行 Harness](https://www.anthropic.com/engineering/effective-harnesses-for-long-running-agents) 中，我们通过初始化智能体、逐功能工作的编码智能体以及会话间上下文重置，解决了多会话编码的连贯性问题。上下文重置是关键解锁：当时 Harness 使用 Sonnet 4.5，它表现出前面提到的“上下文焦虑”。创建一个能在上下文重置下良好工作的 Harness，是让模型保持专注的关键。而 Opus 4.5 基本自行消除了这种行为，因此我得以在这个 Harness 中完全移除上下文重置。所有智能体以连续会话形式运行整个构建过程，[Claude Agent SDK](https://platform.claude.com/docs/en/agent-sdk/overview) 的自动压缩机制处理了上下文增长。

在此基础上，我构建了一个三智能体系统，每个智能体针对先前运行中观察到的具体短板：

**规划器**：早期 Harness 要求用户提前提供详细规格。我希望自动化这一步，于是创建了一个规划器智能体，它接收 1-4 句简单提示并扩展为完整的产品规格。我提示它在范围上保持雄心，并专注于产品上下文和高层次技术设计，而非详细的技术实现（担心如果规划器提前指定过于细粒度的细节并出错，错误会向下游级联）。同时要求规划器寻找在产品规格中融入 AI 功能的机会。（示例见文末附录）

**生成器**：延续早期 Harness 一次一个功能的做法，我指示生成器以“冲刺”方式工作，从规格中逐个挑选功能。每个冲刺使用 React + Vite + FastAPI + SQLite（后期改为 PostgreSQL）栈实现应用，并在冲刺结束时自我评估，然后交给 QA。它还集成了 git 版本控制。

**评估器**：早期 Harness 产出的应用往往看起来很炫，但实际使用时仍有真实 bug。为此，评估器使用 Playwright MCP 以用户方式点击运行中的应用，测试 UI、API 和数据库状态。然后根据发现的 bug 以及一套基于前端实验改编的标准（涵盖产品深度、功能性、视觉设计和代码质量）对每个冲刺评分。每个标准都有硬性阈值，任何一项不达标即判定冲刺失败，生成器会收到详细反馈。

每个冲刺开始前，生成器和评估器会协商“冲刺合同”：在编写任何代码前，先明确这一块工作的“完成”标准。因为产品规格有意保持高层次，我需要这一步来桥接用户故事与可测试实现之间的差距。生成器提出构建内容和验证方式，评估器审查提案并迭代，直到双方达成一致。

通信全部通过文件进行：一个智能体写文件，另一个读取并回复（可在同一文件或新建文件）。生成器根据合同构建后，将工作交给 QA。这种方式既保证忠于规格，又避免过早过度指定实现细节。

### 运行 Harness

第一个版本的 Harness 使用 Claude Opus 4.5，我对相同用户提示分别运行完整 Harness 和单智能体系统进行对比（当时 Opus 4.5 是我们最好的编码模型）。

我使用的提示是生成一个复古视频游戏制作器：

> 创建一个 2D 复古游戏制作器，包含关卡编辑器、精灵编辑器、实体行为和可玩测试模式等功能。

下表显示两种 Harness 的运行时长和总成本：

| **Harness**     | **持续时间** | **成本** |
|-----------------|--------------|----------|
| Solo            | 20 分钟     | $9      |
| Full harness    | 6 小时      | $200    |

完整 Harness 成本高出 20 倍以上，但输出质量差异一目了然。

我原本期待的是一个界面，能让我构建关卡及其组成部分（精灵、实体、瓦片布局），然后点击播放即可实际游玩。打开 solo 运行的结果时，初始应用看起来基本符合预期。

但点击操作后问题接踵而至：布局浪费空间，固定高度面板让大部分视口空荡荡；工作流僵硬——提示我先创建精灵和实体，但 UI 没有任何引导。更关键的是实际游戏无法运行：实体出现在屏幕上却对输入毫无响应。查看代码发现实体定义与游戏运行时的连接完全断开，且界面上没有任何提示。

而完整 Harness 的运行从同一句提示开始，但规划器将其扩展为跨越十个冲刺的 16 项功能规格，远超 solo 运行。除了核心编辑器和播放模式，还包括精灵动画系统、行为模板、音效音乐、AI 辅助精灵生成器和关卡设计师，以及支持导出并分享链接的功能。我还让规划器读取我们的 [前端设计技能](https://github.com/anthropics/claude-code/blob/main/plugins/frontend-design/skills/frontend-design/SKILL.md)，并据此为应用创建视觉设计语言。每个冲刺前，生成器和评估器都会协商详细合同，明确实现细节和验证行为。

该应用立即展现出远超 solo 运行的精致度和流畅性：画布占满整个视口，面板大小合理，界面拥有与规格设计方向一致的视觉身份。solo 运行中的部分笨拙依然存在（例如仍未明确提示用户应先创建精灵和实体再填充关卡），这更多是基础模型的产品直觉差距，而非 Harness 设计要解决的问题，但也暗示 Harness 内可通过针对性迭代进一步提升。

在编辑器中使用时，新运行的优势更加明显。精灵编辑器功能更丰富，工具调色板更干净，颜色选择器更好用，缩放控件也更便捷。

因为我要求规划器在规格中融入 AI 功能，应用还内置了 Claude 集成，能通过提示生成游戏的各个部分，大幅加快工作流。

最大的差异在播放模式：我终于能移动实体并实际玩游戏。虽然物理效果还有些粗糙（角色跳上平台后会出现重叠，感觉不对劲），但核心功能已能正常工作——而 solo 运行完全做不到。游玩一会儿后，我也发现了 AI 关卡构建的一些限制（如遇到无法跳过的巨大墙壁而卡住），这表明 Harness 还能通过常识改进和边缘案例处理进一步优化。

从日志来看，评估器严格将实现控制在规格范围内。每个冲刺，它都会逐一检查合同测试标准，并通过 Playwright 实际操作应用，对任何偏离预期的行为记录 bug。合同非常细粒度——仅第 3 个冲刺就包含 27 条关卡编辑器标准——评估器的发现具体到可以直接行动，无需额外调查。下表是评估器发现的部分问题示例：

| **合同标准**                          | **评估器发现** |
|---------------------------------------|----------------|
| 矩形填充工具允许点击拖拽填充选中瓦片的矩形区域 | **失败** —— 工具仅在拖拽起点/终点放置瓦片，未填充整个区域。fillRectangle 函数存在但 mouseUp 时未正确触发。 |
| 用户可选中并删除已放置的实体生成点   | **失败** —— LevelEditor.tsx:892 的删除键处理程序要求 selection 和 selectedEntityId 同时存在，但点击实体仅设置 selectedEntityId。条件应改为 selection || (selectedEntityId && activeLayer === 'entity')。 |
| 用户可通过 API 重新排序动画帧        | **失败** —— PUT /frames/reorder 路由定义在 /{frame_id} 路由之后。FastAPI 将 'reorder' 匹配为 frame_id 整数，返回 422：“unable to parse string as an integer.” |

让评估器达到这个水平花了不少功夫。开箱即用的 Claude 是一个糟糕的 QA 智能体。早期运行中，它会识别出真实问题，然后自己说服自己“问题不大”而批准通过。它还倾向于浅尝辄止，而非深入探测边缘情况，导致许多细微 bug 漏网。调优方法是阅读评估器日志，找出其判断与我的预期不符的案例，然后更新 QA 提示来解决这些问题。经过几轮这样的开发循环后，评估器的评分才达到我认为合理的水平。即使如此，Harness 输出仍显示出模型 QA 能力的极限：小的布局问题、某些交互不够直观，以及评估器未充分测试的深层嵌套功能中的隐藏 bug。通过进一步调优显然还有更多验证空间。但与 solo 运行（核心功能完全无法使用）相比，提升效果非常明显。

### 对 Harness 的迭代

第一版 Harness 的结果令人鼓舞，但它也笨重、缓慢且昂贵。下一步逻辑是寻找在不降低性能的前提下简化 Harness 的方法。这既是常识，也是更普遍原则的体现：Harness 中的每个组件都编码了对模型自身能力的假设，这些假设值得压力测试——因为它们可能错误，也可能随着模型进步迅速过时。我们的博客文章 [Building Effective Agents](https://www.anthropic.com/research/building-effective-agents) 将核心理念表述为“找到最简单的解决方案，只有在必要时才增加复杂度”，这也是任何维护代理 Harness 的人都会反复遇到的模式。

第一次简化尝试中，我大幅砍掉 Harness 并尝试了一些创意新想法，但未能重现原始性能，同时也难以判断哪些部分真正承载负载。因此我转向更系统的方法：每次移除一个组件，观察对最终结果的具体影响。

在进行这些迭代时，我们也发布了 Opus 4.6，这进一步激励我降低 Harness 复杂度。有充分理由相信 4.6 需要的脚手架比 4.5 少得多。正如我们的 [发布博客](https://www.anthropic.com/news/claude-opus-4-6) 所述：“[Opus 4.6] 规划更谨慎，能更长时间维持代理任务，在更大代码库中运行更可靠，并拥有更好的代码审查和调试能力来发现自身错误。”它在长上下文检索上也有大幅提升。这些正是 Harness 原本用来弥补的能力。

### 移除冲刺结构

我首先彻底移除了冲刺结构。冲刺结构原本帮助模型将工作分解为可控块以保持连贯性。鉴于 Opus 4.6 的改进，有理由相信模型已能原生处理这项工作，无需这种分解。

我保留了规划器和评估器，因为它们仍能带来明显价值。没有规划器，生成器会严重低估范围：直接面对原始提示时，它会未经规划就开始构建，最终产出的功能远少于有规划器的情况。

移除冲刺结构后，我将评估器改为在整个运行结束时进行单次评估，而非每个冲刺评分。由于模型能力大幅提升，评估器在不同任务中的负载作用也发生了变化：其价值取决于任务是否超出模型单独可靠完成的能力边界。在 4.5 上，这个边界很近——我们的构建已处于生成器单独能良好完成的边缘，评估器能在全流程捕捉有意义的 bug。而在 4.6 上，模型原始能力提升，边界向外移动。过去需要评估器检查才能连贯实现的任务，现在很多已进入生成器单独可良好处理的范围，此时评估器就成了不必要的开销。但对于仍处于生成器能力边缘的部分，评估器依然能提供真实提升。

实际含义是：评估器不是固定“是/否”的决定。当任务超出当前模型单独可靠完成的能力时，它才值得投入成本。

在结构简化的同时，我还添加了提示来改进 Harness 如何将 AI 功能融入每个应用，特别是让生成器构建一个能通过工具驱动应用自身功能的真正智能体。这需要真实迭代，因为相关知识较新，Claude 训练数据覆盖较少。但经过足够调优后，生成器已能正确构建智能体。

### 更新后 Harness 的结果

为了测试更新后的 Harness，我使用以下提示生成一个数字音频工作站（DAW），即用于作曲、录音和混音的音乐制作程序：

> 使用 Web Audio API 在浏览器中构建一个功能完整的 DAW。

这次运行依然耗时且昂贵，大约 4 小时，token 成本约 124 美元。

大部分时间花在构建器上，它在没有 Opus 4.5 所需的冲刺分解的情况下连贯运行了两个多小时。

| **智能体 & 阶段**     | **持续时间**   | **成本**   |
|-----------------------|----------------|------------|
| 规划器                | 4.7 分钟      | $0.46     |
| 构建（第 1 轮）       | 2 小时 7 分钟 | $71.08    |
| QA（第 1 轮）         | 8.8 分钟      | $3.24     |
| 构建（第 2 轮）       | 1 小时 2 分钟 | $36.89    |
| QA（第 2 轮）         | 6.8 分钟      | $3.09     |
| 构建（第 3 轮）       | 10.9 分钟     | $5.88     |
| QA（第 3 轮）         | 9.6 分钟      | $4.06     |
| **V2 Harness 总计**   | **3 小时 50 分钟** | **$124.70** |

与之前一样，规划器将单句提示扩展为完整规格。从日志看，生成器模型在规划应用和智能体设计、连接智能体以及移交 QA 前进行测试方面做得很好。

尽管如此，QA 智能体仍能捕捉到真实差距。在第一轮反馈中，它指出：

> 这是一个设计保真度优秀、AI 智能体扎实、后端良好的强大应用。主要失败点是“功能完整性”——虽然应用看起来很炫且 AI 集成工作正常，但几个核心 DAW 功能仅为展示而缺乏交互深度：时间线上无法拖动/移动片段，没有乐器 UI 面板（合成器旋钮、鼓垫），也没有视觉效果编辑器（EQ 曲线、压缩器仪表）。这些不是边缘情况，而是让 DAW 真正可用的核心交互，规格中明确要求了它们。

第二轮反馈又捕捉到几个功能差距：

> 剩余差距：
> - 音频录制仍为存根（按钮可切换，但无麦克风捕获）
> - 片段边缘拖拽调整大小和分割功能未实现
> - 效果可视化仅为数字滑块，而非图形化（无 EQ 曲线）

生成器独自工作时仍容易遗漏细节或将功能做成存根，而 QA 仍在捕捉这些“最后一公里”问题供生成器修复方面发挥了价值。

根据提示，我期待的是一个能创建旋律、和声、鼓点，将它们编排成歌曲，并一路获得集成智能体帮助的程序。下面的视频展示了结果。

## 下一步是什么

随着模型持续进步，我们大致可以预期它们能工作更长时间、处理更复杂的任务。在某些情况下，这意味着围绕模型的脚手架重要性会随时间降低，开发者只需等待下一个模型就能解决某些问题。但另一方面，模型越强大，可开发出超越模型基线能力的复杂任务的 Harness 空间就越大。

基于此，这项工作有几点经验值得继续沿用：始终用你正在构建的模型进行实验，阅读它在真实问题上的轨迹，并调优其性能以达到预期结果；在处理更复杂任务时，有时通过任务分解并为每个方面应用专门智能体能获得额外空间；当新模型发布时，通常最好重新审视现有 Harness，剥离那些不再承载性能负载的部分，并加入新组件以实现此前不可能达到的更大能力。

通过这项工作，我坚信：随着模型进步，有趣的 Harness 组合空间不会缩小，而是会移动。AI 工程师的有趣工作，就是持续发现下一个新颖组合。
