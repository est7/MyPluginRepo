# 真实场景：半自动 Agent 开发工作流

> 面向已有代码库的真实迭代场景。
> 目标不是让 Agent 全自动开发，而是让程序员保留边界定义、纠偏和验收权，Agent 负责高频分析与实现。

---

## 适用范围

适合：

- 已有代码库上的中小需求迭代
- 有明确功能入口的页面、模块或接口改动
- 程序员可以做最终 E2E / 真机 / 远端环境验证的团队

不适合直接套最简版：

- 0 到 1 的系统级架构设计
- 跨多个子系统的大改造
- 高安全、高合规、高并发风险模块
- 团队约定高度隐式、文档严重缺失的老系统

---

## 核心原则

1. 程序员主导关键决策，Agent 主导高频展开。
2. 先审理解，再审方案，再审执行结果。
3. 没有明确入口、影响范围和验证方式，不进入编码。
4. 需求、方案、执行、验收是不同层级，delta 必须按层级重入。
5. 已确认的基线 Plan 默认不可变，增量变化以 Delta 追加记录。

---

## 推荐流程

### Phase A：需求结构化

1. 产品提供原始 PRD
2. Agent 转成实现 PRD
3. Agent 发起需求澄清
4. 程序员补充并确认实现 PRD

实现 PRD 至少应包含：

- 用户故事或 BDD / Gherkin 场景
- 主路径
- 边界条件与异常场景
- 数据约束与非功能约束
- `In Scope / Out of Scope / Non-goals`

### Phase B：代码理解与方案确认

5. 程序员提供最小实现上下文
6. Agent 沿入口定向探索代码库
7. Agent 输出代码理解摘要
8. 程序员确认代码理解是否正确
9. Agent 输出结构化 Plan
10. 程序员修正并确认 Plan

最小实现上下文至少包括：

- 一个真实代码入口
- 一个验收锚点

可选补充：

- 相关模块名
- 设计稿或对照行为
- API 文档
- 已知约束和团队约定

### Phase C：实现与验收

11. Agent 按 Plan 执行，并补齐约定测试
12. 程序员做轻量工程质量审查
13. 程序员做 E2E / 真机 / UI 验证
14. 按 delta 类型分流重入
15. 验收通过，结束

轻量工程质量审查只检查四类问题：

- 是否违反既有架构边界
- 是否引入不必要依赖或新模式
- 核心改动是否带必要测试
- 是否出现明显坏味道

---

## 五个关键优化点

### 1. Step 6 后必须有代码理解摘要

不要让 Agent 探索完代码后直接出 Plan。

代码理解摘要至少写清：

- 入口与主调用链
- 关键模块及职责
- 数据流 / 状态流
- 影响范围初判
- 假设与不确定点

这样程序员先审“理解对不对”，再审“方案对不对”。

### 2. Delta loop 必须定义重入点

不是所有问题都直接回灌给 Agent。

- 新需求、范围变化、验收标准变化：回需求澄清或实现 PRD 确认
- 代码理解错误、影响范围判断错误：回代码理解摘要或 Plan
- 方案偏差、UI 方向问题：回 Plan
- 实现 bug、小修、小范围测试补充：直接回执行

### 3. Plan 要固定 schema

至少包含：

- 修改目标
- 影响范围
- 实现思路
- 风险项
- 验证方式
- 测试计划
- 不确定点

没有固定格式，Plan review 会退化成凭感觉判断。

### 4. Step 9 和 Step 10 之间要有轻量质量 gate

E2E 只能验证功能，不会自动拦住结构性技术债。

这个 gate 不做完整 code review，只拦：

- 架构越层
- 不必要依赖
- 明显坏味道
- 漏测试

### 5. 测试必须显式进入流程

不要只靠 E2E 兜底。

Agent 在 Plan 阶段就要标注：

- 哪些点要补单测
- 哪些点适合集成测试
- 哪些只能靠 E2E 验证

## Delta 处理策略

### 不建议的做法

- 每次 delta 都原地重写整份 Plan
- 永远只追加，不整理当前有效视图

前者会丢掉已确认决策、增加 token 消耗，并放大全局漂移；后者会让语义冲突和审查成本持续堆积。

### 推荐模型

把 Plan 拆成三层：

1. `Baseline Plan`
2. `Delta Log`
3. `Effective View`

含义：

- `Baseline Plan`：程序员确认后的基线版本，默认不可变
- `Delta Log`：每次增量变化的追加记录
- `Effective View`：当前执行视图，按需汇总基线与增量

这更接近一种受控的 patch log / event sourcing 模型。

### Delta 的最小字段

- `delta_id`
- `type`
- `reason`
- `impact`
- `decision`
- `task_patch`
- `status`

### 何时只追加 Delta

满足以下全部条件时，只追加，不改基线：

- 不新增一级模块
- 不改变主路径
- 不改变验收标准
- 不改变 baseline 风险判断
- 不推翻已有 task 依赖

### 何时生成新的 Effective View

满足任一条件时，生成新的当前有效视图：

- 影响范围新增模块
- task 依赖关系变化
- 已确认决策被显式否定
- 多条 delta 持续作用于同一模块
- 程序员已无法快速判断当前该按什么执行

### 何时重新进入 Plan Gate

满足任一条件时，重新做 Plan 确认：

- 需求范围变化
- 主调用链理解错误
- 架构层决策变化
- 引入新依赖、新协议或新状态模型
- 验收标准变化

一句话规则：

> Baseline Plan 一经确认默认不可变；后续变化以 Delta Decision 追加记录。仅当 delta 改变主路径、影响范围、依赖关系或验收标准时，才生成新的 Effective Plan 并重新进入 Plan Gate。

## 最小模板

### 1. 程序员提供的最小上下文包

- 功能入口
- 验收锚点
- 可选：相关模块、设计稿、API 文档、已知约束

### 2. 代码理解摘要模板

```md
## 入口与主调用链
- 入口：
- 主路径调用链：
- 关键分支：

## 模块与职责
- 模块 A：
- 模块 B：

## 数据流 / 状态流
- 谁产生：
- 谁持有：
- 谁消费：
- 谁更新：

## 影响范围初判
- 直接影响：
- 间接影响：

## 假设 / 不确定点
- 假设：
- 未确认项：
```

### 3. Plan 模板

```md
## 修改目标

## 影响范围
- [模块] -> [新增 / 修改 / 删除]
- [调用方] -> [是否受影响]

## 实现思路

## 风险项
- [风险] -> [触发条件] -> [应对]

## 测试与验证
- 单测：
- 集成测试：
- UI / E2E：

## 不确定点
```

---

## 一句话定义

这套流程本质上不是“提需求后 Agent 自动做完”，而是“以程序员为裁决者的受控代码生成流程”。
